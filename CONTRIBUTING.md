## ü§ù How Can I Contribute?

### Reporting Bugs

Before creating bug reports, please check existing issues. When you create a bug report, include as many details as possible:

**Bug Report Template:**
```markdown
**Describe the bug**
A clear and concise description of what the bug is.

**To Reproduce**
Steps to reproduce the behavior:
1. Go to '...'
2. Click on '....'
3. See error

**Expected behavior**
What you expected to happen.

**Screenshots**
If applicable, add screenshots.

**Environment:**
 - Home Assistant Version: [e.g. 2026.2.0]
 - Node-RED Version: [e.g. 3.1.0]
 - Browser: [e.g. chrome, safari]

**Additional context**
Any other context about the problem.
```

### Suggesting Enhancements

Enhancement suggestions are tracked as GitHub issues. When creating an enhancement suggestion, include:

- **Clear title** describing the enhancement
- **Detailed description** of the proposed functionality
- **Why this enhancement would be useful** to most users
- **Possible implementation** if you have ideas

### Pull Requests

**Process:**

1. Fork the repo
2. Create your feature branch (`git checkout -b feature/AmazingFeature`)
3. Commit your changes (`git commit -m 'Add some AmazingFeature'`)
4. Push to the branch (`git push origin feature/AmazingFeature`)
5. Open a Pull Request

**Guidelines:**

- Update documentation to reflect changes
- Add screenshots if UI/UX changes
- Test thoroughly before submitting
- Follow existing code style
- One feature per PR
- Reference relevant issues

## üìù Coding Standards

### Node-RED Flows

- Use descriptive node names
- Add comments to complex function nodes
- Group related nodes visually
- Use consistent color coding

### JavaScript

- Use modern ES6+ syntax
- Add comments for complex logic
- Handle errors gracefully
- Use meaningful variable names

### Documentation

- Use clear, concise language
- Include code examples
- Add screenshots where helpful
- Keep markdown properly formatted

## üîí Security & Privacy

**NEVER commit:**
- Actual credentials
- Authentication tokens
- Personal information
- School names or student data
- Real cookie values
- API keys

**Always:**
- Use placeholder values
- Blur sensitive info in screenshots
- Test with sanitized data
- Document security considerations
